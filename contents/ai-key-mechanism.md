# AI 핵심 메커니즘 이해

'도구'가 아닌 '원리'와 '흐름'에 집중하기

> 새로운 라이브러리(LangChain, LlamaIndex 등)나 프레임워크는 계속해서 쏟아집니다. 이를 다 배우는 것은 불가능에 가깝습니다. 대신, 그 저변에 깔린 핵심 원리를 파악하는 것이 중요합니다.

# AI 개발 및 활용의 핵심 기둥이 되는 5가지 핵심 메커니즘

이 5가지 흐름만 파악해도, 새로운 AI 도구가 나올 때마다 "아, 이건 결국 그 원리를 쓴 거네?"라고 바로 간파할 수 있습니다. 코딩을 몰라도 개념과 맥락을 이해하는 것이 핵심입니다.

---

## 1. LLM의 작동 본질: "AI는 사실 '다음 단어 맞추기' 게임을 한다"

ChatGPT나 Claude 같은 AI가 글을 쓰는 방식은 생각보다 단순한 원리에서 출발합니다. "지금까지 나온 말을 보고, 다음에 올 가장 자연스러운 단어를 확률적으로 고른다." 이것을 수십억 개의 문장으로 반복 학습한 결과가 오늘날의 AI입니다.

| 개념 | 쉬운 설명 | 실생활 비유 |
| --- | --- | --- |
| **Tokenization (토큰화)** | AI는 글자를 그대로 읽지 않고, 의미 단위의 '조각(Token)'으로 쪼개 숫자로 변환해 처리 | 악보를 통째로 외우는 게 아니라 음표 하나하나로 쪼개서 읽는 것 |
| **Context Window (문맥 창)** | AI가 한 번의 대화에서 기억할 수 있는 분량의 한계. 이 범위를 벗어난 내용은 잊어버림 | 독서 중 책갈피 앞뒤 몇 페이지만 기억하고 나머지는 흐릿해지는 것 |
| **Temperature (온도)** | 답변이 얼마나 '창의적'이고 '다양'할지 조절하는 값. 낮으면 정확하고 일관되며, 높으면 엉뚱하고 창의적 | 낮은 온도 = 사전 그대로 받아쓰기, 높은 온도 = 자유 작문 |

> **왜 중요한가?** 한국어가 영어보다 API 비용이 더 드는 이유와 대화가 길어질수록 AI가 앞 내용을 잊어버리는 이유가 모두 여기서 비롯

---

## 2. RAG (검색 증강 생성): "AI에게 '참고자료'를 실시간으로 건네주기"

AI는 학습 시점 이후의 정보를 모릅니다. 사내 문서나 최신 데이터도 모릅니다. RAG는 이 한계를 극복하는 가장 현실적인 방법으로, 현재 기업용 AI 서비스의 90% 이상이 이 방식을 사용합니다.

**작동 흐름 (3단계)**

1️⃣. 사용자가 **질문을 입력**

2️⃣. AI가 수만 개의 문서 중 **질문과 관련된 문서를 빠르게 탐색** (**Retrieval**).

3️⃣. 찾은 문서를 질문과 함께 AI에게 전달해 **답변을 생성** (**Generation**).

| 개념 | 쉬운 설명 | 실생활 비유 |
| --- | --- | -- |
| **Embedding** | 텍스트를 수치(좌표)로 변환해 의미가 비슷한 문장끼리 가까운 위치에 모아두는 기술 | 도서관에서 비슷한 주제의 책을 같은 서가에 모아두는 분류 체계 |
| **Vector Database** | 임베딩된 수만 개의 문서를 저장하고, 질문과 가장 유사한 문서를 즉시 찾아주는 저장소 | 사서가 "이 질문엔 이 책이 딱이에요"라고 즉시 골라주는 것 |

> **왜 중요한가?** LangChain, LlamaIndex 같은 도구들은 결국 이 RAG 흐름을 편리하게 구현하는 도구입니다. 흐름을 알면 어떤 도구도 낯설지 않습니다.

---

## 3. Agentic Workflow: "AI가 스스로 판단하고 행동하는 단계"

단순히 질문에 답하는 것을 넘어, AI가 스스로 목표를 분석하고 → 필요한 도구를 골라 실행하고 → 결과를 보고 다음 행동을 결정하는 방식입니다.

**예시: "이번 달 매출 보고서 만들어줘"**

1. 데이터베이스 조회 도구 실행 → 데이터 수집
2. 계산 도구 실행 → 합계·비율 계산
3. 문서 작성 도구 실행 → 보고서 초안 생성
4. 결과 검토 후 오류 발견 시 → 스스로 재시도

| 개념 | 쉬운 설명 |
| --- | ---- |
| **Function Calling**   | "날씨 알려줘"라는 요청에 AI가 직접 날씨 앱을 실행하고 결과를 가져오는 능력 |
| **Chain of Thought** | 복잡한 문제를 한 번에 풀지 않고, 단계를 나눠 차근차근 생각하게 하는 방식 | **Planning & Feedback Loop** | 계획 → 실행 → 결과 확인 → 수정의 반복 구조로, 틀리면 스스로 다시 시도 |

> **왜 중요한가?** 요즘 화제인 'AI 에이전트'나 'AI 자동화'는 모두 이 원리를 기반으로 합니다. AI가 단순 답변자에서 '일을 처리하는 조수'로 진화하는 핵심입니다.

---

## 4. Fine-tuning vs Prompting: "AI를 바꿀 것인가, AI에게 잘 설명할 것인가"

AI를 내 목적에 맞게 활용하는 방법은 크게 두 가지입니다.

| 구분 | 설명 | 비유 | 비용/난이도 |
| --- | --- | --- | --- |
| **Fine-tuning** | AI 모델 자체의 지식과 말투를 업데이트. 특정 분야 전문 언어, 기업 고유 스타일 등을 내재화할 때 사용 | 신입사원을 수개월 교육해 회사 방식을 체득시키는 것 | **높음** (데이터·비용·시간 필요) |
| **Prompt Engineering** | AI 자체는 건드리지 않고, 질문과 지침을 잘 구성해 원하는 답을 유도 | 시험 당일 참고자료를 잘 정리해 오픈북 시험을 보게 하는 것 | **낮음** (즉시 시도 가능) |

> **💡 Note:** 최근 트렌드는 대부분의 경우 Fine-tuning보다 좋은 데이터를 프롬프트로 잘 넣어주는 **(RAG + Prompt Engineering)** 방식이 더 빠르고 효율적입니다. Fine-tuning은 꼭 필요한 경우에만 선택하는 '최후의 수단'에 가깝습니다.

---

## 5. Evaluation (평가): "AI가 잘 대답했는지 어떻게 확인하는가?"

AI 서비스를 만들 때 가장 어렵고, 그만큼 중요한 단계입니다. 사람이 직접 모든 답변을 검토할 수 없기 때문에, AI가 AI를 채점하는 구조가 많이 쓰입니다.

| 개념 | 쉬운 설명 | 예시 |
| -- | --- | --- |
| **LLM-as-a-Judge** | 더 성능이 좋은 AI 모델에게 다른 AI의 답변을 채점 | GPT-4o에게 "이 답변이 정확한지 0~10점으로 평가해줘" |
| **Hallucination Check** | AI가 지어낸 답변(거짓 정보)인지, 실제 근거 문서에 기반한 답변인지 검증 | 참고문헌 없이 논문을 인용하면 표절·조작으로 잡아내는 것 |
| **Faithfulness** | AI의 답변이 제공된 자료 범위 안에서 나온 것인지 확인하는 지표 | "이 내용이 실제로 문서에 있나?"를 자동으로 체크 |

> **왜 중요한가?** AI가 아무리 말을 잘 해도, 그 말이 맞는지 검증하지 않으면 신뢰할 수 없습니다. 특히 의료·법률·금융처럼 정확성이 중요한 분야에서는 이 단계가 전체 품질을 결정합니다.

---

# 5가지 메커니즘 한눈에 보기

```text
[사용자 질문 입력]
       ↓
① 'LLM 본질': AI가 언어를 어떻게 이해하고 생성하는가
       ↓
② 'RAG': 필요한 정보를 외부에서 찾아 AI에게 전달
       ↓
③ 'Agent': 단순 답변을 넘어 도구를 사용해 실제 작업 수행
       ↓
④ 'Fine-tuning / Prompting': AI를 목적에 맞게 최적화
       ↓
⑤ 'Evaluation': 결과가 신뢰할 수 있는지 검증
```
